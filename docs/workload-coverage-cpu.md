# Workload coverage & perf scoreboard (CPU / PoCL)

Diverse AI + scientific-computing workloads run through our OpenCL VM on PoCL (`JAX_PLATFORMS=opencl PJRT_OCL_DEVICE=Portable`, the host-dispatch engine: no spin-barrier, clFinish/ring-drain per phase, `-DVMO_CPU_TILES` float8 EW + packed SGEMM) vs the reference native JAX CPU/XLA backend (`JAX_PLATFORMS=cpu`) on the same AMD Ryzen 9 3900X (12c/24t). Generated by `tools/bench_suite/run_suite.py --cpu`. Each backend runs in its own subprocess; latency is median-of-5-rounds, adaptive iters, after warmup. `gap = ours_ms / cpu_ms` (lower is better; <1 means we beat XLA-CPU). PoCL has no TF32, so correctness is checked f32-tight (`allclose atol=rtol=1e-3`).

**Coverage: 16/18 workloads run on our backend (88%).**

## Host-dispatch overhead analysis (perf attack, 2026-07-21)

On CPU we run the **host-dispatch engine** (the persistent spin-barrier megakernel
is a data race on PoCL — no co-resident workgroups — so it is disabled). Each
schedule **phase** (= one global barrier boundary) becomes one
`clEnqueueNDRangeKernel` of `vm2_seg`. The seg-tab **ring** (§12 follow-up) already
batches phases: they are enqueued back-to-back on the in-order queue with **one**
non-blocking seg-tab write per group and a `clFinish` **only** at while-cond reads,
ring wrap (256 slots), and program end. Measured with `PJRT_OCL_PHASE_STATS=1`:

| workload | phases = kernel enqueues | drains (clFinish) | ours ms | µs / enqueue |
|---|---|---|---|---|
| rk4_ode | 2501 | 10 | 189.0 | 75.6 |
| spring_mass | 721 | 3 | 61.4 | 85.1 |
| heat2d | 390 | 2 | 46.1 | 118 |
| lstm | 245 | 1 | 75.3 | 307 |
| gru | 242 | 1 | 72.0 | 298 |
| logistic_map | 201 | 1 | 17.0 | 84 |
| hh_neuron | 201 | 1 | 173.8 | 865 |
| monte_carlo | 74 | 1 | 38.3 | 518 |
| transformer | 68 | 1 | 277.4 | 4080 |
| attention | 12 | 1 | 27.2 | 2264 |
| mlp | 6 | 1 | 16.8 | 2800 |
| cnn | 4 | 1 | 3.7 | 933 |
| layernorm | 3 | 1 | 2.7 | 894 |

**The sync is already fully batched — it is not the bottleneck.** logistic_map runs
200 loop iterations with **one** `clFinish` for the whole Execute (the ring holds all
201 phases). rk4_ode's 10 drains are pure ring-wrap (2501 phases / 256). Removing them
saves nothing measurable: the in-order queue is throughput-bound on the enqueues, and a
wrap-drain only waits for work that had to run anyway. So "batch phases between clFinish"
is exhausted.

**The wall is the per-phase kernel *enqueue*, and it is a fixed PoCL launch floor.**
Two controlled sweeps:

- *Grid-size independent.* A barrier-free elementwise `fori_loop` costs ~78 µs/iter at
  **B=256** and ~86 µs/iter at **B=65536** (256× the data) — the launch dominates, the
  work is free.
- *Workgroup-count independent.* logistic_map is ~16 ms at `PJRT_OCL_VM_LANES` = 2, 4, 8,
  **and** 24. Fewer/more workgroups per launch does not move it.

So each `clEnqueueNDRangeKernel` on PoCL is a ~75–85 µs fixed dispatch cost we cannot
shrink from our side (it matches §12's "PoCL launch floor 17–52 µs", now ~80 µs for the
seg kernel). The **only** lever is the *number* of enqueues. Two cost regimes fall out of
the µs/enqueue column:

1. **Enqueue-count-bound (the loop laggards):** rk4/spring_mass/logistic_map sit right at
   the ~75–85 µs floor — every µs is launch overhead, `phases ≈ loop_iters × body_phases`.
   These are 1200–4700× behind because XLA-CPU compiles the whole `lax.scan`/`fori_loop`
   into **one** native function with **zero** per-iteration dispatch. hh_neuron (865
   µs/enqueue) is the same regime plus a transcendental-heavy body (many `exp`/`div` tiles
   PoCL scalarizes).
2. **Throughput-bound (the matmul graphs):** transformer/mlp/attention/lstm have *few*
   phases (6–68) but thousands of µs each — that time is real SGEMM/reduce work, our
   in-VM matmul vs XLA's Eigen/MKL, not launch overhead.

### Why the single-program design does *not* beat XLA-CPU here (the thesis, answered)

The megakernel's premise — "one program, no per-op dispatch" — was supposed to win exactly
this loop/small-op regime. It does not, for two independent reasons:

- **XLA-CPU has no per-op dispatch to beat.** On a GPU, native JAX pays a kernel launch per
  fused op; on CPU, XLA lowers the *entire* graph (loop included) to one compiled function.
  There is no dispatch tax on the reference side, so our "avoid dispatch" advantage is worth
  0×.
- **Our host-dispatch still pays a launch per phase.** Because PoCL has no safe device-wide
  barrier, a phase boundary *is* a kernel-launch boundary. A T-iteration loop with a P-phase
  body costs `T·P` launches × ~80 µs — the very per-op cost the design set out to avoid,
  reintroduced by the portability fallback.

**The structural cut** is barrier elision for **lane-local** loops (§10a, designed, not
built): when a loop body has no cross-lane data flow (each lane owns a disjoint,
iteration-stable slice — true for pure-elementwise carries like logistic_map/hh_neuron, false
for spring_mass/heat2d neighbour stencils and lstm/gru matmuls), *all* its barriers are
spurious and the whole `T·P`-phase loop can run in **one** `vm2_seg`-style launch where each
workgroup runs its lane's flattened entry stream. That collapses logistic_map from 201 → 1
enqueue. **But even the ideal single-launch floor (~80 µs + output readback + clFinish ≈
150–200 µs) is still ~10× XLA-CPU's 14 µs** for the tiniest loops — PoCL's launch floor is
above XLA-CPU's *total* runtime for anything under ~80 µs. It is correctness-critical
(a mis-certified cross-lane loop races) so it stays a gated PoC, not a default-path change,
per the PoC-first rule.

### Bottom line for CPU

- **0 / 16 workloads reach ≤1× XLA-CPU**; the closest is cnn at 7.2×, then layernorm 12×,
  attention 16×. On this 12-core part XLA-CPU (LLVM/Eigen native code) runs most workloads
  in **single-digit-to-tens of µs** — below our single-launch floor — so beating it is not
  reachable with the host-dispatch engine regardless of enqueue count.
- Where we are *only* enqueue-bound (the scan/loop laggards, 190–4700×), the fix is real and
  identified (barrier elision → one launch), and it would move them from ~80 µs × iters down
  to a single launch — a 10–100× local win — but not under 1×.
- **Correctness is the win to bank:** every one of the 16 passers is f32-tight vs XLA-CPU
  (max rel ≤ 3e-3, most ≤ 1e-6) — PoCL has no TF32, so the backend is bit-honest here in a
  way the NVIDIA/TF32 path is not.
